# WORKFLOW.md — Флоу изменений после релиза

Сквозной процесс любого изменения в Recenza **после закрытия 12 фаз**: от идеи до прода и записи
в журнал. Обязателен для каждой сессии Claude Code, которая пишет код. Дополняет (не заменяет):
`CLAUDE.md` (архитектура + краткий деплой), `ENVIRONMENTS.md` §6 (runbook сервера),
`TESTING.md` (устройство тест-слоя), `PLAN.md` § «Пост-релизные итерации» (журнал).

> Фазовые ритуалы `PROMPT.md`/`PLAN.md` относились к миграции и остаются историей.
> Для новых изменений действует ЭТОТ документ.

---

## 0. Классификация изменения (выбирает объём процесса)

| Класс | Примеры | Тесты | Цикл качества |
|---|---|---|---|
| **S — мелочь** | текст/стили одного компонента, правка доки, конфиг без поведения | новых НЕ нужно; полный e2e прогнать | build+lint+e2e; сабагенты не обязательны |
| **M — поведение** | новый UI-элемент, правка логики роута, фикс бага с изменением поведения | обновить затронутые спеки; новый e2e-кейс, если появилось НОВОЕ поведение | + профильный сабагент по зоне (см. §3) |
| **L — фича/поверхность** | новый роут/API/таблица, изменение схемы БД, новая страница | обязательный новый спек (happy-path + гейтинг + негатив); при схеме — миграция | полный флот сабагентов, как у фаз |

Сомневаешься между классами — бери больший. Изменение схемы БД всегда L.

## 1. Старт: запись и ветка

1. Прочитай `CLAUDE.md` (конвенции + гочи) и хвост `PLAN.md` § «Пост-релизные итерации».
2. **Ветка от свежего `main`**: `hotfix-<slug>` (фикс/мелочь) или `feature-<slug>` (класс L).
   В `main` напрямую не коммитить — push в него = автодеплой на прод.
3. Для класса L — **запись в `PLAN.md`** (журнал итераций) ДО начала: дата, ветка, цель,
   план тестов. Для S/M запись делается по факту в том же PR (см. §5).

## 2. Разработка

- Соблюдай жёсткие конвенции `CLAUDE.md` (Drizzle без raw SQL, Unix seconds, `ulid()`,
  `parseJson`, params-Promise, русский UI, токены вместо raw-цветов и т.д.).
- Миграции: правка `src/lib/db/schema.ts` → `npm run db:generate` → **ручное ревью SQL**
  (гоча drizzle-kit: composite-PK/onDelete) → `npm run test:reset` для применения на тест-БД.
  Миграции только аддитивные/only-forward (откат кода на проде не откатывает БД).
- UI сверяй с прототипом (`docs/prototype/ui_kits/blog/src/**`) и `DESIGN-TOKENS.md`.

## 3. Тесты (см. классификацию §0)

- Автотесты живут в `testing/e2e/**` (конвенции — `TESTING.md` и шапки спеков):
  локаторы по роли/тексту, seed-константы из `helpers/seed.ts`, мутирующие спеки —
  `serial` + `reseed()` в beforeAll/afterAll, негативные API — через `api()`-фикстуру,
  флак-паттерны — идемпотентный `toPass`-ретрай и `waitForResponse` при оптимистичном UI.
- Прогон: **полный** `npx playwright test` локально (118+ должны быть зелёными, 0 skip;
  ⚠️ cron.spec требует стенд, поднятый самим Playwright, либо `CRON_SECRET` в `.env.test`).
- Новый e2e-кейс обязателен, если появилось новое пользовательское поведение или новый API-гейт.

## 4. Цикл качества (блокирующий гейт перед PR)

Всегда: `npm run lint` чистый + полный e2e зелёный (build проверит CI).
Дополнительно по классу/зоне изменения:

| Зона | Сабагент/скилл |
|---|---|
| auth / api / секреты / загрузки | `security-reviewer` (+скилл `security-checklist`) — 0 критических |
| любой нетривиальный код (M/L) | `code-reviewer` — нет P0/P1 |
| UI-компоненты/стили | `design-watcher` — нет P0 (+визуальный чек через Playwright MCP: light/dark, 375px) |
| публичные страницы/metadata | `seo-optimizer` |

P0/P1 чинятся до PR; P2/P3 — в backlog журнала.

## 5. PR → прод (конвейер автоматический)

1. Коммит(ы) → `git push -u origin <ветка>` → `gh pr create` (описание: что/зачем, чем проверено).
2. PR-гейт CI `e2e-smoke` (lint+build+@smoke; `.env.test` генерится сам). Красный из-за
   инфраструктуры GitHub («Runner not acquired») — просто `gh run rerun`.
3. Зелёный → `gh pr merge --squash --delete-branch`.
4. Push в `main` **сам** запускает `deploy.yml`: standalone-сборка → rsync на VPS → миграции →
   симлинк-релиз → рестарт → health-check (детали/откат — `ENVIRONMENTS.md` §6 и CLAUDE.md
   § «Деплой изменений»). Дождись `success`: `gh run list --workflow=deploy`.
5. **Проверка на проде**: `curl -I https://recenza.ru` = 200 + точечная проверка именно твоего
   изменения (страница/эндпоинт). UI — глазами или Playwright MCP по прод-URL (read-only,
   мутирующие сценарии на проде запрещены).
6. **Запись в журнал**: секция `### <дата> · PR #<N> <ветка>` в `PLAN.md`
   § «Пост-релизные итерации» — что сделано, находки, backlog. Если изменение затронуло
   архитектуру/конвенции/гочи — обнови `CLAUDE.md` (и `ENVIRONMENTS.md`, если сервер).

## 6. Чек-лист закрытия итерации

- [ ] Полный e2e зелёный локально; smoke зелёный в CI
- [ ] Цикл качества по классу пройден (P0/P1 = 0)
- [ ] PR смержен squash-ем, ветка удалена
- [ ] `deploy` = success, изменение проверено на https://recenza.ru
- [ ] Журнал в `PLAN.md` дополнен; CLAUDE.md/ENVIRONMENTS.md актуальны (если задеты)
- [ ] Секретов в диффе нет (`git status` перед коммитом; `.env*`, `*.db`, ключи — никогда)

## Экстренный случай (прод сломан)

Ветка `hotfix-<slug>` с максимальным приоритетом; если чинить дольше, чем откатить —
сначала откат релиза на сервере (симлинк + restart, `ENVIRONMENTS.md` §6.4), потом фикс по флоу.
