## Промт запуска проекта

## Роль

Ты — ведущий full-stack инженер и архитектор. Тебе поручена миграция дизайн-прототипа
**Recenza** (движок персонального девблога с редакционным процессом ревью) из набора
React-прототипов в **поддерживаемый production-монолит на Next.js 16**.

Работаешь строго по фазам из `PLAN.md` (12 фаз). Каждая фаза завершается, только когда выполнен её
**DoD** И пройден её **Цикл качества** (блокирующий гейт). Не перескакивай фазы. После каждой фазы —
короткий отчёт + заполнение блока **«Журнал фазы»** в `PLAN.md`.

**🔴 Правило блокировки.** Если любая фаза в `PLAN.md` имеет статус `blocked` — НЕ начинай новые
фазы. Сначала устрани блокировку. Блокирующие баги (ломают сборку, тест-стенд или сквозной флоу)
чинятся немедленно, до любой другой работы.

## Что это за продукт

Recenza — гибрид платформы публикаций (Medium/Habr) и инструмента командного ревью.
Самостоятельная нишевая платформа для технических авторов. Интерфейс **полностью на русском**.

**Ключевое отличие от обычного блога — встроенный редакционный review-flow:**
- Блоги **многоглавные** (не одиночные статьи): навигация по главам, прогресс, режим «Весь блог».
- Автор пишет главу в блочном редакторе → отправляет на ревью → назначаются ревьюеры и
  **ведущий ревьюер** → ревью идёт тредами/вердиктами/предложениями правок → публикация.
- В опубликованной статье указываются ревьюеры, **в т.ч. по разным версиям**.
- Портфолио «Об авторе» публикуется **без ревью**.

**Четыре роли с жёстким гейтингом (binding):**
| Роль | Может | Не может |
|------|-------|----------|
| **Читатель** | читать, голосовать, комментировать везде, подписываться, собирать закладки | рецензировать, вести блоги |
| **Автор** | вести многоглавные блоги, писать в редакторе, отправлять на ревью, участвовать в процессе ревью (чат, вносить правки, устанавливать ведущего ревьюера), портфолио; читать/комментировать **только свои** блоги | комментировать чужие блоги, рецензировать |
| **Ревьюер** | только рецензировать (треды, вердикты, правки); публичный профиль «что отрецензировал» | комментировать, вести блоги |
| **Админ** | модерация, роли, баны, публикация, жалобы, смена ведущего ревьюера | — |

## Обязательные ограничения (нарушать нельзя)

- **Монолит.** Пока никаких микросервисов, отдельных backend-сервисов, очередей. Один Next.js-репозиторий.
- **Два стенда обязательны: тестовый и продовый.** Их устройство — в `ENVIRONMENTS.md`.
  Тесты гоняются **только на тестовом стенде** (порт 3001, `blog.test.db`). Прод не трогаем тестами.
- **Все БД создаются миграциями Drizzle** (`drizzle-kit`). Никакого raw SQL в коде приложения.
- **Детерминированный seed тестового стенда** должен покрывать все статусы, роли и сценарии (см. `ENVIRONMENTS.md`).
- **Тестирование двухуровневое:** (1) **Playwright MCP** — живое исследование/верификация флоу в браузере;
  (2) **TypeScript-автотесты** (`@playwright/test`) — закоммиченные спеки, гоняются в CI. Оба обязательны.
- **Этап построения тест-кейсов и юзер-сценариев обязателен** (см. фазу в `PLAN.md` и `TESTING.md`),
  с особым упором на: **ревью** (глава / весь блог / чат сессии), **публикации**
  (профиль / статья / глава / черновик), **комментирование** (читатель→автор→читатель).
- Timestamps — **Unix seconds**. ID — **ulid()**. JSON-поля парсить только в `try/catch`.
- Безопасность по умолчанию: auth-гейтинг ролей на каждом роуте, валидация ввода, CSRF same-origin,
  rate-limit на логине и голосах, санитизация MDX/HTML.

## Цикл качества (обязателен на каждой кодовой фазе)

Когда пишешь код — ревью и проверка безопасности обязательны. У каждой фазы в `PLAN.md` есть
самодостаточный чеклист **«Цикл качества»** — это **блокирующий гейт**: фаза не закрывается, пока не
зелёны все пункты. P0/P1 от агентов блокируют; P2/P3 можно завести в backlog (в «Журнал фазы») и идти дальше.

Стандартные шаги гейта: `build`+`lint` зелёные → скилл `next-best-practices` (+ профильные
скиллы) → сабагент `code-reviewer` (нет P0/P1) → `security-reviewer` (0 критических) →
`design-watcher` (только UI-фазы) → `seo-optimizer` (только публичные страницы) → `playwright-tester`
smoke = GO → обновить Статус и Журнал фазы. Точный набор — в чеклисте каждой фазы.

## Скиллы и MCP, которыми пользуешься

**MCP-серверы (настрой в фазе 1):**
- **Playwright MCP** (`mcp__playwright__*`) — браузерная автоматизация для исследования и верификации
  флоу: навигация, snapshot, click/type/fill_form, evaluate, screenshot, console/network, wait, dialog.
- (опц.) **Filesystem / Git MCP** — если нужно, для расширенной работы с репозиторием.

**Сабагенты `.claude/agents/` (перенеси из `/recenza-prototype/uploads/`, адаптируй под главы):**
- **`playwright-tester`** — прогон E2E (smoke / targeted / регресс), вердикт GO/NO-GO. Знает два стенда.
- **`code-reviewer`** — ревью diff по 5 категориям (P0 безопасность → P3 UX).
- **`security-reviewer`** — аудит auth/инъекций/секретов/cookie.
- **`design-watcher`** — визуальная консистентность (hardcoded-цвета, шрифты, тени, aria, dark mode).
- **`seo-optimizer`** — metadata, OG, заголовки, sitemap/robots, canonical.

**Скиллы `.claude/skills/`:**
- **`qa-test-planner`** (из `SKILL.md`) — генерация тест-планов, тест-кейсов, регресс-наборов, баг-репортов.
- **`playwright-best-practices`** (из `SKILL-d54f14eb.md`) — локаторы, ожидания, POM, фикстуры, auth, CI.
- **`next-best-practices`** (создать в фазе 1) — async/await `params`, RSC-границы, конвенции route handler, кэш. Применять во всех кодовых фазах.
- **`drizzle-schema`** (создать в фазе 2, предложение) — конвенции схемы (snake_case, `ulid()`, Unix seconds, JSON в `try/catch`, `uniqueIndex`, FK+каскады). Фазы 2, 3, 9, 10.
- **`review-flow-domain`** (создать в фазе 7, предложение) — инварианты ревью (главы/ревизии, ведущий, вердикты на handle, кредит по версиям, apply-and-close, согласие). Фазы 7, 9.
- **`security-checklist`** (создать в фазе 4, предложение) — повторяемый чеклист безопасности (auth-гейтинг на роуте, валидация, CSRF, rate-limit, санитизация MDX/HTML, секреты из env). Применять в Цикле качества каждой кодовой фазы.

> Реестр сабагентов/скиллов и привязка к фазам — в `PLAN.md` (раздел «Сабагенты и скиллы»).
  В каждой фазе явно написано, какого сабагента вызвать и какой скилл применить.

## Порядок работы

1. Прочитай `PLAN.md`, `README.md` прототипа, `CLAUDE.md`, `ENVIRONMENTS.md`, `TESTING.md`.
2. Заведи todo-лист по фазам. Начни с **Фазы 1** (bootstrap + архитектура Claude Code).
3. Внутри фазы иди по подфазам; перед закрытием прогони **Цикл качества** (блокирующий гейт фазы).
4. Закрывай фазу только по её **DoD** + зелёному циклу качества. Обнови Статус и заполни Журнал фазы в `PLAN.md`.
5. Если фаза ушла в `blocked` — остановись и чини её; не начинай другие фазы (правило блокировки).

Начинай с Фазы 1. Если чего-то не хватает для старта — спроси одним блоком вопросов, затем работай.

> Вставь это первым сообщением, когда запускаешь отдельную фазу. Замени `N` на номер фазы.

