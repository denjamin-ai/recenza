# TC-FLOWS — Сквозные мультиролевые сценарии (Recenza, Фаза 11.1)

**Кейсов: 16** (P0: 11 · P1: 5).
**Легенда приоритетов:** P0 — security/data-loss/критический инвариант/основной путь публикации · P1 — основная фича роли сломана без workaround · P2 — частичная деградация с workaround · P3 — косметика.

Покрытие акцентных сценариев `TESTING.md` §3: REV-CHAPTER, REV-WHOLE-BLOG, REV-SESSION-CHAT, REV-PRIMARY, REV-VERSIONS, PUB-DRAFT, PUB-CHAPTER-V2, PUB-ARTICLE, PUB-PORTFOLIO, COM-THREAD, COM-STALE, COM-EDIT-WINDOW, COM-GATING + матчинг Фазы 9: MATCH-INVITE, MATCH-RECRUIT, MATCH-BOARD.

## Общие предусловия (для всех кейсов)

- Тест-стенд **:3001** (`npm run dev:test`), БД `blog.test.db`, seed выполнен (`npm run test:reset`). Прод/`:3000` не трогать.
- Пароль всех seed-пользователей: `password`. Вход пользователей — `/login`, кнопка «Войти».
- Админ НЕ в seed-пользователях: вход на `/admin/login` паролем из env `ADMIN_PASSWORD_PLAIN` (`POST /api/auth`).
- Мультиролевые шаги выполнять в **разных браузерных контекстах/профилях** (по одному на роль) — сессии не должны затирать друг друга.
- Сценарии **мутируют БД**: перед каждым сквозным прогоном — свежий `npm run test:reset` (если в Preconditions кейса не сказано иное).
- Синхронизация статусов между экранами — через обновление страницы или поллинг 30 с (вебсокетов нет); «мгновенно» ожидать нельзя.

## REV-CHAPTER: Полный цикл ревью одной главы — от черновика до публикации

**Priority:** P0
**Type:** E2E / Functional (мультиролевой)
**Роли-участники:** автор (`author`), ревьюер (`reviewer`)

### Objective
Проверить основной путь публикации: submit черновика → приглашение (pending) → accept ревьюером → тред с suggestion → «Применить и закрыть» автором → «Одобрить» → публикация; кросс-экранная синхронизация статуса (инбокс/кабинет/ридер).

### Preconditions
- Тест-стенд :3001, seed выполнен (`npm run test:reset`).
- Глава `chp_draft` (slug `generators`) в статусе `draft`, ревьюеры не назначены.
- Два браузерных контекста: автор и ревьюер.

### Test Steps
1. **Автор:** войти на `/login` (`author` / `password`), нажать «Войти».
   **Expected:** редирект в кабинет; меню aria-label «Меню пользователя» показывает Антона Автора.
2. **Автор:** открыть редактор `/author/blog/async-deep-dive/generators/edit`.
   **Expected:** редактор главы «Генераторы и итераторы», статус «Черновик», кнопка «Отправить на ревью →».
3. **Автор:** нажать «Отправить на ревью →»; в шторке «Отправить на ревью» задать навыки статьи `Event Loop`, `React`; в пикере подбора выбрать «Раиса Ревьюер» (`reviewer`) ведущей; отправить.
   **Expected:** ревизия v1 переходит в `under-review` («На ревью»); создано приглашение `review_invitations` со статусом `pending`; в `chapter_reviewers` записи НЕТ (ревью не стартовало); повторная попытка правки главы в редакторе блокируется (API → 409).
4. **Ревьюер:** войти (`reviewer` / `password`), открыть инбокс `/reviewer`.
   **Expected:** карточка входящего приглашения по главе «Генераторы и итераторы»: match%, навыки, кнопки «Принять» / «Отклонить».
5. **Ревьюер:** нажать «Принять».
   **Expected:** приглашение `accepted`; глава появляется среди активных ревью; `review_load` Раисы 2→3 (занятость `full`); автор после обновления видит согласие.
6. **Ревьюер:** открыть `/reviewer/review/chp_draft`, создать тред на блоке текста с suggestion (предложенная правка формулировки).
   **Expected:** тред в статусе `open`, bauble-маркер у блока, suggestion отображается в треде.
7. **Автор:** открыть review-страницу главы (`/author/blog/async-deep-dive/generators/review`), в треде нажать «Применить и закрыть».
   **Expected:** текст блока обновлён **in-place** (новая ревизия НЕ создана), тред → `resolved`, счётчик открытых тредов уменьшился.
8. **Ревьюер:** обновить страницу (или дождаться поллинга 30 с).
   **Expected:** блок показывает применённый текст, тред отмечен решённым; bauble↔thread sync корректен.
9. **Ревьюер:** нажать «Одобрить».
   **Expected:** вердикт `approve` зафиксирован; прогресс вердиктов 1/1.
10. **Автор:** обновить review-страницу, нажать «Опубликовать».
    **Expected:** ревизия v1 → `published`; в `reviewer_history` кредит Раисе за v1; `review_load` Раисы −1.
11. **Все роли:** проверить синхронизацию: инбокс `/reviewer` — глава «Опубликована»; кабинет `/author` — статус главы обновлён; ридер `/blog/async-deep-dive/generators` — глава открывается гостю, `<title>` = «Генераторы и итераторы».
    **Expected:** статус согласован на всех трёх экранах (после обновления страниц).

### Test Data
- Глава: `chp_draft` (blog `async-deep-dive`, slug `generators`).
- Навыки статьи: `Event Loop`, `React` (совпадают с компетенциями `reviewer` → высокий match; занятость `busy` 2/3 — выбирается).
- Пользователи: `author`, `reviewer` (пароль `password`).

### Post-conditions
- `chp_draft` опубликована (v1), запись в `reviewer_history`, приглашение `accepted`.
- БД мутирована — `npm run test:reset` перед следующим сценарием.

### Notes
- Согласие (Фаза 9): submit создаёт ТОЛЬКО `review_invitations` — ревью-гейты (тред/вердикт/publish) должны быть недоступны ревьюеру до «Принять».
- Смежные кейсы: PUB-ARTICLE (продолжение), MATCH-INVITE (акцент на подборе), PUB-DRAFT (обход через force-approve).

## REV-WHOLE-BLOG: Ревью всего блога — независимые статусы глав и навигация по strip

**Priority:** P0
**Type:** E2E / Functional (мультиролевой)
**Роли-участники:** автор (`author`), ревьюер (`reviewer`)

### Objective
Проверить работу ReviewPage на блоге с главами во всех статусах: переключение глав через strip (`role="tablist"`), режим «Весь блог», независимость статусов и сохранение контекста навигации.

### Preconditions
- Тест-стенд :3001, seed выполнен (`npm run test:reset`).
- Блог `async-deep-dive` с 4 главами в разных статусах (published / under-review / changes-requested / draft).

### Test Steps
1. **Автор:** войти (`author`), открыть `/author/blog/async-deep-dive`.
   **Expected:** 4 главы с независимыми статусами: «Цикл событий» — опубликована; «Промисы изнутри» — на ревью; «Async/await на практике» — нужны правки; «Генераторы и итераторы» — черновик.
2. **Автор:** открыть review-страницу «Промисы изнутри» (`/author/blog/async-deep-dive/promises/review`).
   **Expected:** ReviewPage со strip глав (`role="tablist"`), активная вкладка — «Промисы изнутри»; у каждой вкладки статус-пилл (цвет + текст, не только цвет).
3. **Автор:** переключиться в strip на «Async/await на практике».
   **Expected:** контекст сменился: тред `thr_resolved_1` (resolved), вердикты v1 — `lena_review` «Нужны правки», `reviewer` «Одобрено»; URL и заголовок обновились.
4. **Автор:** вернуться на «Промисы изнутри».
   **Expected:** контекст не потерян: открытые треды `thr_open_1`, `thr_open_2`, чат сессии, прогресс вердиктов (0 approve из 2 активных; у `lena_review` — «Нужны правки»).
5. **Автор:** включить режим «Весь блог».
   **Expected:** сводка/прогресс ревью по всем главам блога; статусы соответствуют шагу 1 и независимы друг от друга.
6. **Ревьюер:** войти (`reviewer`), открыть `/reviewer/review/chp_under_review`.
   **Expected:** доступ есть (назначен); навигация показывает только назначенные ревьюеру главы (`promises`, `async-await`); черновик `generators` в ревью-контекст не попадает.
7. **Ревьюер:** переключиться на «Async/await на практике» и обратно.
   **Expected:** треды/вердикты каждой главы изолированы (resolved-тред `async-await` не «протекает» в `promises`); прогресс каждой главы корректен.

### Test Data
- Главы: `chp_published` (event-loop), `chp_under_review` (promises), `chp_changes` (async-await), `chp_draft` (generators).
- Треды: `thr_open_1`, `thr_open_2` (promises), `thr_resolved_1` (async-await).

### Post-conditions
- Read-only сценарий: состояние БД не изменено (мутаций нет).

### Notes
- A11y-инвариант: strip глав обязан иметь `role="tablist"` (правило frontend-design).
- Регресс-ловушка: смена вкладки не должна терять локальный UI-стейт из-за Suspense-remount (см. gotcha про `startTransition`).

## REV-SESSION-CHAT: Чат ревью-сессии — несколько ревьюеров и автор, изоляция от тредов

**Priority:** P0
**Type:** E2E / Functional (multi-user, 3 контекста)
**Роли-участники:** ревьюер (`reviewer`), автор (`author`), ревьюер (`lena_review`)

### Objective
Проверить чат сессии (`review_chat`) на главе `chp_under_review`: сообщения видны всем участникам сессии, чат не смешивается с тредами, presence отображается.

### Preconditions
- Тест-стенд :3001, seed выполнен; три браузерных контекста.
- В seed чат `chp_under_review` содержит `rch_1`–`rch_3`.

### Test Steps
1. **Ревьюер (reviewer):** войти, открыть `/reviewer/review/chp_under_review`, открыть панель чата сессии.
   **Expected:** 3 seed-сообщения в хронологическом порядке: «Начинаю смотреть главу.» (Раиса) → «Спасибо! Жду замечаний.» (автор) → «Подключилась к ревью.» (Лена).
2. **Ревьюер (reviewer):** отправить сообщение «Проверил примеры кода — вопросов нет.».
   **Expected:** сообщение появляется в чате отправителя сразу, с автором «Раиса Ревьюер».
3. **Автор:** войти (второй контекст), открыть review-страницу `promises`, панель чата.
   **Expected:** видны все 4 сообщения (новое — после загрузки страницы/поллинга 30 с).
4. **Автор:** ответить «Спасибо, жду вердикта.».
   **Expected:** сообщение добавлено в чат (5 сообщений).
5. **Ревьюер (lena_review):** войти (третий контекст), открыть `/reviewer/review/chp_under_review`.
   **Expected:** все 5 сообщений видны третьему участнику сессии.
6. **Все:** сверить треды `thr_open_1`/`thr_open_2` и чат.
   **Expected:** чат-сообщения НЕ появляются в тредах, реплики тредов НЕ появляются в чате (разные хранилища `review_chat` ≠ `threads/thread_replies`).
7. **Любой участник:** проверить presence-индикаторы участников.
   **Expected:** Раиса отмечена online (seed `chapter_reviewers.online=true`); presence статичен (без live-обновления — вебсокетов нет).

### Test Data
- Глава `chp_under_review` (promises); чат `rch_1`, `rch_2`, `rch_3`.
- Пользователи: `reviewer`, `author`, `lena_review`.

### Post-conditions
- В `review_chat` добавлены 2 сообщения — сброс seed перед следующим прогоном.

### Notes
- Multi-user тест: в Playwright — несколько `browserContext` в одном спеке.
- Негативный смежный кейс (в TC-REVIEWER/TC-READER): не-участник сессии не имеет доступа к чату (`resolveReviewAccess`).

## REV-PRIMARY: Смена ведущего ревьюера и кросс-экранное распространение

**Priority:** P0
**Type:** E2E / Functional (мультиролевой)
**Роли-участники:** автор (`author`), ревьюеры (`reviewer`, `lena_review`), админ

### Objective
Проверить смену ведущего (primary) на главе `chp_under_review` автором и распространение изменения на инбоксы ревьюеров, кабинет автора и админ-очередь.

### Preconditions
- Тест-стенд :3001, seed выполнен.
- `chp_under_review` (promises): ведущий — `reviewer`; принявший ревьюер `lena_review`; в seed есть pending-запрос `pcr_1` (reviewer → lena_review).

### Test Steps
1. **Автор:** войти, открыть `/author/blog/async-deep-dive/promises/review`.
   **Expected:** ведущий — «Раиса Ревьюер»; в action-bar доступна кнопка «Сменить ведущего»; виден контекст запроса на смену (`pcr_1`), если он отображается в UI.
2. **Автор:** нажать «Сменить ведущего».
   **Expected:** модалка с aria-label «Сменить ведущего ревьюера»; кандидаты — только принявшие ревьюеры главы (в списке есть «Лена Базы»).
3. **Автор:** выбрать `lena_review`, подтвердить.
   **Expected:** ведущий сменён: чип primary у «Лена Базы»; у Раисы бейдж ведущего снят; тост об успехе без перезагрузки страницы (нет потери UI-стейта).
4. **Ревьюер (reviewer):** войти, открыть инбокс `/reviewer`.
   **Expected:** глава «Промисы изнутри» осталась в активных, но БЕЗ пометки «ведущий».
5. **Ревьюер (lena_review):** войти, открыть `/reviewer`.
   **Expected:** глава помечена как «ведущий» (primary) у Лены.
6. **Админ:** войти на `/admin/login`, открыть `/admin/review`.
   **Expected:** в очереди ревью глава `promises` показывает ведущего `lena_review`; админу также доступна собственная смена ведущего.
7. **Автор:** открыть кабинет `/author/blog/async-deep-dive`.
   **Expected:** детали главы показывают нового ведущего — изменение распространилось кросс-экранно.

### Test Data
- Глава `chp_under_review`; ведущий до: `reviewer`, после: `lena_review`; запрос `pcr_1` (pending).
- Пользователи: `author`, `reviewer`, `lena_review`, админ (`ADMIN_PASSWORD_PLAIN`).

### Post-conditions
- `chapters.primaryHandle` = `lena_review` — сброс seed перед следующим прогоном.

### Notes
- Смена ведущего доступна автору и админу; ревьюер может только запросить (`primary_change_requests`).
- Кандидаты — только из `chapter_reviewers` (принявшие); пригласённый, но не ответивший `sergey_review` в списке быть не должен.

## REV-VERSIONS: Кредит ревьюеров по версиям — текущие чипами, прошлые за раскрытием

**Priority:** P0
**Type:** Functional / UI (read-only)
**Роли-участники:** гость (достаточно), читатель

### Objective
Проверить, что опубликованная глава с двумя ревизиями показывает ревьюеров текущей версии (v2) чипами, а ревьюеров прошлой версии (v1) — за раскрытием; кредит сохраняется в профилях ревьюеров.

### Preconditions
- Тест-стенд :3001, seed выполнен.
- `chp_published` (event-loop): v1 published (ревьюеры `reviewer`+`lena_review`), v2 published (ревьюеры `reviewer`+`max_review`), `prev_blocks` у v2 заполнен.

### Test Steps
1. **Гость:** открыть `/blog/async-deep-dive/event-loop`.
   **Expected:** рендерится контент v2 («Расширенная версия с диаграммами и примерами»); `<title>` содержит «Цикл событий».
2. **Гость:** найти блок ревьюеров главы.
   **Expected:** чипы ревьюеров ТЕКУЩЕЙ версии (v2): «Раиса Ревьюер» (ведущий) и «Макс Девопс»; «Лена Базы» среди текущих НЕ отображается.
3. **Гость:** раскрыть элемент «прошлые версии» (disclosure/details).
   **Expected:** кредит v1: «Раиса Ревьюер», «Лена Базы» — виден только за раскрытием.
4. **Гость:** открыть профиль `/u/reviewer`.
   **Expected:** раздел «что отрецензировал» включает «Цикл событий» (кредит за обе версии не задублирован в UI).
5. **Гость:** открыть профиль `/u/lena-review`.
   **Expected:** кредит за v1 «Цикла событий» сохранился, хотя в v2 Лена не участвовала.

### Test Data
- `chp_published`, ревизии `rev_pub_1` (v1) / `rev_pub_2` (v2); `reviewer_history`: v1 — reviewer+lena_review, v2 — reviewer+max_review.

### Post-conditions
- Read-only: БД не изменена.

### Notes
- Источник данных — `reviewer_history` (кредит по версиям), не текущий состав `chapter_reviewers`.
- Смежный кейс: PUB-CHAPTER-V2 (живой прогон обновления `reviewer_history`).

## PUB-DRAFT: Публикация черновика через admin force-approve (обход гейта all-approve)

**Priority:** P0
**Type:** E2E / Functional + Negative (мультиролевой)
**Роли-участники:** автор (`author`), админ; ревьюер (`sergey_review`) — пассивно

### Objective
Проверить запасной путь публикации: автор не может опубликовать без вердиктов (409), админ публикует главу через «Force-approve (опубликовать)» в обход гейта «все approve».

### Preconditions
- Тест-стенд :3001, seed выполнен.
- `chp_draft` (generators) в статусе `draft`.

### Test Steps
1. **Автор:** войти, открыть редактор `generators`, нажать «Отправить на ревью →»; оставить seed-навыки (`Генераторы`, `Итераторы`), через «Показать всех» пригласить `sergey_review`; отправить.
   **Expected:** v1 → `under-review`; приглашение `pending`; ревьюер НЕ активен (в `chapter_reviewers` пусто).
2. **Автор:** открыть review-страницу `generators`.
   **Expected:** кнопка «Опубликовать» отсутствует/неактивна — нет ни одного `approve`.
3. **Автор (негатив):** выполнить прямой API-запрос публикации главы (POST publish по `chp_draft`).
   **Expected:** **409** — гейт «все approve» перепроверяется на сервере, публикация отклонена.
4. **Админ:** войти на `/admin/login` (пароль из `ADMIN_PASSWORD_PLAIN`), открыть `/admin/review`.
   **Expected:** «Генераторы и итераторы» в очереди ревью, прогресс вердиктов 0.
5. **Админ:** нажать «Force-approve (опубликовать)», подтвердить в двухшаговом подтверждении.
   **Expected:** глава опубликована в обход гейта; статус в очереди обновлён.
6. **Гость:** открыть `/blog/async-deep-dive/generators` и ленту `/`.
   **Expected:** глава читается; блог поднялся в ленте с новой главой.
7. **Автор:** открыть кабинет `/author`.
   **Expected:** статус главы — «Опубликована».

### Test Data
- Глава `chp_draft` (generators); приглашение → `sergey_review` (останется без ответа).
- Пользователи: `author`, админ.

### Post-conditions
- `chp_draft` published (v1) без записей вердиктов ревьюеров; pending-приглашение Сергея потеряло актуальность.
- Сброс seed перед следующим прогоном.

### Notes
- Негативный шаг 3 закрывает часть матрицы §2: «Публиковать главу — автор ✅ только при approve».
- Админ НЕ создаёт блоги/главы — force-approve публикует существующую ревизию, не контент админа.

## PUB-CHAPTER-V2: Публикация новой версии главы — снапшот, сброс вердиктов, reviewer_history

   **Expected:** приглашение `accepted`; `review_load` 0→1; глава в активных ревью.
6. **Ревьюер:** открыть ревью главы, нажать «Одобрить».
   **Expected:** approve 1/1 — гейт публикации открыт.
7. **Автор:** обновить review-страницу, нажать «Опубликовать».
   **Expected:** глава published; `review_load` Сергея −1; в кабинете появился блок оценки ревьюеров.
8. **Автор:** поставить Сергею оценку 5★.
   **Expected:** оценка сохранена (`reviewer_ratings`); отметка «сохранено»; повторная правка оценки не создаёт дубликата.
9. **Гость (приватность):** открыть `/u/sergey-review`.
   **Expected:** виден ТОЛЬКО агрегат (обновлённый `reviewerRating`, счётчик оценок +1); индивидуальная оценка «5★ от author» нигде не отображается.
10. **Ревьюер:** проверить свой кабинет/профиль.
    **Expected:** свою полученную оценку видит сам ревьюер (и админ) — приватный контур соблюдён.

### Test Data
- Глава `chp_draft`; навыки: `Безопасность`, `Криптография`; ревьюер `sergey_review` (3.8★, n=3, 0/3 free).
- Seed-эталоны приватных оценок: `reviewer` 5★, `max_review` 4★ (за `chp_published`).

### Post-conditions
- `generators` published; агрегат рейтинга Сергея пересчитан; сброс seed перед следующим прогоном.

### Notes
- Компетенции ревьюера ≠ навыки статьи — разные сущности; навыки главы видны читателю в ридере (aria-label «Ключевые навыки главы»).
- Флаг «навыки не совпадают» (match <50%) — отдельный негативный кейс в TC-REVIEWER (эталон seed `inv_flagged`).

## MATCH-RECRUIT: Нет совпадений → recruit-запрос админу → approve → направление на публичной доске

**Priority:** P1
**Type:** E2E / Functional (мультиролевой)
**Роли-участники:** автор (`author`), админ, гость

### Objective
Проверить запасной путь при отсутствии подходящих ревьюеров: автор видит пустой подбор и шлёт запрос админу; админ одобряет; направление публикуется на публичной доске `/board`.

### Preconditions
- Тест-стенд :3001, seed выполнен.
- `chp_draft` с seed-навыками `Генераторы`, `Итераторы` (не совпадают ни с одним ревьюером); в seed уже есть `rec_pending` (author, generators), `rec_approved`, `rec_rejected`.

### Test Steps
1. **Автор:** войти, открыть SubmitSheet главы `generators` с seed-навыками.
   **Expected:** подбор пуст/все совпадения 0% — предлагается запасной путь «запрос админу»; существующий `rec_pending` отображается как «на рассмотрении» (повторная отправка дубля недоступна или создаёт новый запрос — зафиксировать фактическое поведение).
2. **Админ:** войти на `/admin/login`, открыть `/admin/recruit`.
   **Expected:** запрос `rec_pending` (от `author`, навыки «Генераторы», «Итераторы») со статусом «на рассмотрении»; в истории — одобренный (`rec_approved`) и отклонённый с причиной (`rec_rejected`).
3. **Админ:** на `rec_pending` нажать «Одобрить».
   **Expected:** статус → «одобрен»; создано/опубликовано направление на публичной доске.
4. **Гость:** открыть `/board`.
   **Expected:** доска «Ищем ревьюеров» содержит новое направление с навыками «Генераторы», «Итераторы» рядом с seed-направлениями `bc_frontend` (React/TypeScript, hot) и `bc_backend` (Node.js/SQL).
5. **Автор:** проверить свой кабинет/уведомления.
   **Expected:** автору виден вердикт «одобрен» по его запросу.
6. **Админ (альтернативная ветка, справочно):** «Отклонить с причиной» на другом запросе.
   **Expected:** статус «отклонён» + причина, доступная автору (эталон — `rec_rejected`).

### Test Data
- Запросы: `rec_pending` (generators), `rec_approved`, `rec_rejected`; доска: `bc_frontend`, `bc_backend`.
- Пользователи: `author`, админ, гость.

### Post-conditions
- `rec_pending` → approved; на доске новое направление; сброс seed перед следующим прогоном.

### Notes
- Блог нельзя опубликовать без ревью — recruit-запрос лишь запасной путь поиска ревьюеров, публикацию он не открывает.
- Негативные кейсы доступа к `/admin/recruit` (не-админ → 307/403) — в TC-ADMIN/TC-GUEST.

## MATCH-BOARD: Заявка с публичной доски → админ принимает → выдача роли reviewer с переносом компетенций

**Priority:** P1
**Type:** E2E / Functional (мультиролевой)
**Роли-участники:** гость, читатель (`reader`), админ

### Objective
Проверить apply-to-review: подача заявки с публичной доски (гостевая и от читателя), приём заявки админом с выдачей роли `reviewer` и переносом компетенций из заявки.

### Preconditions
- Тест-стенд :3001, seed выполнен.
- Доска: `bc_frontend` (hot), `bc_backend`; заявки: `app_user` (от `reader`, pending, навыки CSS/React), `app_guest` (гостевая, accepted).

### Test Steps
1. **Гость:** открыть `/board`.
   **Expected:** публичная доска «Ищем ревьюеров»: `bc_frontend` (React/TypeScript, метка hot), `bc_backend` (Node.js/SQL); форма заявки доступна без входа.
2. **Гость:** заполнить и отправить гостевую заявку (имя/контакт, направление Frontend, навыки).
   **Expected:** заявка создана со статусом «pending»; подтверждение в UI.
3. **Админ:** войти на `/admin/login`, открыть раздел заявок в `/admin/recruit`.
   **Expected:** видна свежая гостевая заявка и seed-заявка `app_user` (от `reader`, «Хочу рецензировать статьи по фронтенду.», CSS/React, pending); у гостевой — «Принять» (без выдачи роли), у заявки пользователя — «Принять (выдать роль)».
4. **Админ:** на `app_user` нажать «Принять (выдать роль)».
   **Expected:** заявка `accepted`; пользователю `reader` выдана роль `reviewer`; компетенции перенесены из заявки (CSS, React).
5. **Читатель→ревьюер:** перелогиниться (`reader` / `password`), открыть `/reviewer`.
   **Expected:** инбокс ревьюера доступен без 307 (роль читается из БД на каждый запрос, не из cookie); приглашений пока нет — корректное пустое состояние.
6. **Гость:** открыть `/u/reader`.
   **Expected:** профиль отображается как ревьюерский («что отрецензировал» — пусто), компетенции CSS/React видны.
7. **Бывший читатель (гейт-инверсия, негатив):** выполнить POST `/api/comments` к `event-loop`.
   **Expected:** **403** — с ролью `reviewer` комментирование запрещено (см. COM-GATING).

### Test Data
- Заявки: `app_user` (reader, pending), `app_guest` (accepted, эталон гостевой); доска: `bc_frontend`, `bc_backend`.
- Пользователи: гость, `reader`, админ.

### Post-conditions
- Роль `reader` изменена на `reviewer` — **обязателен** `npm run test:reset` перед любым другим кейсом с читателем.

### Notes
- Смена роли — только этим админ-путём: обычным API роль не меняется (binding-инвариант).
- Кнопка «Стать ревьюером» живёт в промо-баннере ленты `pb_recruit` (internal → `/board`) — вход на доску проверяется также из карусели.
