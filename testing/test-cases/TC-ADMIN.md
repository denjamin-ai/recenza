# TC-ADMIN — Тест-кейсы: Администратор

Роль: **админ** (env-based, НЕ в таблице `users`; вход паролем `ADMIN_PASSWORD_PLAIN` из `.env.test` через `POST /api/auth`, страница `/admin/login`).
Скоуп: логин/логаут, полноэкранный админ-портал, дашборд, модерация пользователей, жалобы, ревью-очередь (force-approve / смена ведущего / снятие ревьюера), recruit-запросы и заявки, доска «Ищем ревьюеров», баннеры, пожертвования + негативные инварианты.

**Кейсов: 22** (P0: 10 · P1: 9 · P2: 3 · P3: 0; негативных: 8)

Легенда приоритетов: **P0** — security/data-loss/критический инвариант/основной путь публикации · **P1** — основная фича роли сломана без workaround · **P2** — частичная деградация с workaround · **P3** — косметика.

> ⚠️ Все кейсы выполняются ТОЛЬКО на тест-стенде **http://localhost:3001** (`blog.test.db`, `.env.test`). Прод/дев `:3000` не трогать. Перед прогоном — `npm run test:reset`. Кейсы TC-ADMIN-09/10/11 мутируют одну и ту же главу `chp_under_review`, а TC-ADMIN-12/13 и TC-ADMIN-14/15 — одни и те же seed-строки: каждому из этих кейсов нужен свежий seed (`npm run test:reset`).

## TC-ADMIN-01: Вход администратора через /admin/login и выход «Выйти к блогу»

**Priority:** P0
**Type:** Functional

### Objective
Проверить основной auth-путь админа: вход только по паролю (`POST /api/auth`), попадание в полноэкранный портал, корректное завершение сессии.

### Preconditions
- Тест-стенд :3001, seed выполнен (`npm run test:reset`).
- Браузер без активных сессий (cookies `blog_session` очищены).
- Известен пароль админа: значение `ADMIN_PASSWORD_PLAIN` из `.env.test`.

### Test Steps
1. Открыть `http://localhost:3001/admin/login`.
   **Expected:** Страница «Вход администратора» с единственным полем «Пароль» и кнопкой «Войти как администратор». Полей логина/handle нет. Страница не индексируется (meta robots noindex).
2. Отправить форму с пустым полем «Пароль».
   **Expected:** Ошибка «Введите пароль.», запрос на сервер не уходит.
3. Ввести пароль из env `ADMIN_PASSWORD_PLAIN`, нажать «Войти как администратор».
   **Expected:** `POST /api/auth` → 200; переход на `/admin`, далее серверный redirect на `/admin/dashboard`; отображается экран «Сводка».
4. Проверить cookie.
   **Expected:** Установлена cookie `blog_session` (httpOnly).
5. В сайдбаре нажать «Выйти к блогу».
   **Expected:** `DELETE /api/auth` → 200; переход на `/` (публичная лента), сессия завершена.
6. Ввести в адресную строку `http://localhost:3001/admin/dashboard`.
   **Expected:** Серверный редирект (307) на `/admin/login` — сессии больше нет.

### Test Data
- Пароль: env `ADMIN_PASSWORD_PLAIN` (`.env.test`).
- URL: `/admin/login`, `/admin/dashboard`.

### Post-conditions
- Сессия админа завершена; состояние БД не изменено.

### Notes
- Инвариант `SessionData`: `isAdmin` и `userId` не сосуществуют — после входа `GET /api/auth/user` возвращает `user: null` (админ не является `PublicUser`).
- Связанные: TC-ADMIN-02 (rate-limit), TC-ADMIN-21 (307 без прав).

## TC-ADMIN-02: Rate-limit логина админа — 6-я неудачная попытка → 429

**Priority:** P0
**Type:** Security (негативный)

### Objective
Проверить, что логин админа ограничен 5 неудачными попытками за 15 минут (инвариант TESTING.md §4 «Rate-limit логина»).

### Preconditions
- Тест-стенд :3001, seed выполнен (`npm run test:reset`).
- Dev-сервер тест-стенда перезапущен недавно ИЛИ по данному IP не было неудачных логинов за последние 15 минут (rate-limit — in-memory).

### Test Steps
1. Открыть `/admin/login`, ввести неверный пароль `wrong-password`, нажать «Войти как администратор». Повторить всего 5 раз.
   **Expected:** Каждая из попыток 1–5: `POST /api/auth` → 401, в форме ошибка «Неверный пароль.».
2. Выполнить 6-ю попытку с тем же неверным паролем.
   **Expected:** `POST /api/auth` → **429**; в форме сообщение «Слишком много попыток. Попробуйте через 15 минут.».
3. Сразу же попытаться войти с ПРАВИЛЬНЫМ паролем (env `ADMIN_PASSWORD_PLAIN`).
   **Expected:** Запрос также отклоняется 429 — лимит действует независимо от корректности пароля.

### Test Data
- Неверный пароль: `wrong-password`.
- Верный пароль: env `ADMIN_PASSWORD_PLAIN`.

### Post-conditions
- IP заблокирован для логина на 15 минут. Сброс: перезапуск `npm run dev:test` (лимитер in-memory) или ожидание окна.

### Notes
- Лимит считается по `x-forwarded-for` (`src/lib/rate-limit.ts`, 5/15 мин). Для API-уровневой автоматизации изолировать кейс разными значениями `X-Forwarded-For`.
- Выполнять кейс ПОСЛЕДНИМ в сессии ручного прогона (блокирует вход) либо сразу перезапускать стенд.

## TC-ADMIN-05: Тумблер комментирования troll — «Разрешить комментарии» / «Запретить комментарии»

**Priority:** P0
**Type:** Functional / Security (включает негативную проверку 403)

### Objective
Проверить, что админ снимает и возвращает ограничение `commentingBlocked`, и что ограничение реально гейтит публичные комментарии (403). Акцент **COM-GATING** (`commenting_blocked → 403`).

### Preconditions
- Тест-стенд :3001, seed выполнен (`npm run test:reset`).
- Окно A: вход админом. Окно B (другой браузер/профиль): вход пользователем troll (пароль `password`).

### Test Steps
1. Окно B (troll): открыть ридер `/blog/async-deep-dive/event-loop` и попытаться отправить комментарий «Проверка гейта» через форму комментариев.
   **Expected:** Публикация отклонена — `POST /api/comments` → **403** (комментирование запрещено), комментарий не появляется.
2. Окно A (админ): открыть `/admin/users/troll`, нажать «Разрешить комментарии».
   **Expected:** Без ошибок; после обновления кнопка меняется на «Запретить комментарии».
3. Окно B (troll): обновить страницу главы и снова отправить комментарий «Проверка гейта».
   **Expected:** Комментарий публикуется и появляется в списке под главой.
4. Окно A (админ): нажать «Запретить комментарии».
   **Expected:** Кнопка возвращается в состояние «Разрешить комментарии».
5. Окно B (troll): попытаться отправить ещё один комментарий.
   **Expected:** Снова **403**; ранее опубликованный комментарий из шага 3 остаётся видимым (запрет — на новые мутации, не ретроактивное удаление).

### Test Data
- Пользователь: troll / `password` (reader, `commentingBlocked=true` в seed).
- Глава: chp_published (`/blog/async-deep-dive/event-loop`).

### Post-conditions
- `commentingBlocked=true` восстановлен (как в seed); добавлен 1 комментарий troll — рекомендуется `npm run test:reset` перед зависимыми кейсами.

### Notes
- Гейтинг перепроверяется сервером в каждом comments-роуте (`commentGate`), UI-блокировка формы вторична — проверять именно ответ API.
- Связанный инвариант TESTING.md §4: «Гейтинг ролей».

## TC-ADMIN-06: Soft-бан/разбан ghost — скрытие блога из выдачи и недействительность входа

**Type:** Functional / Security

### Objective

   **Expected:** Живая сессия гаснет (self-heal `getCurrentUser`) — пользователь становится гостём; мутационные API отвечают 401.
8. Админ: проверить `/admin/users`.
   **Expected:** ghost по-прежнему в списке (бан — soft, запись пользователя не удалена).

- Пользователь: ghost / `password` (author, `isBlocked=true` в seed).
- Блог: blog_ghost (slug `hidden-blog`).

### Post-conditions
- Состояние seed восстановлено (ghost снова заблокирован).

## TC-ADMIN-07: Hard-delete пользователя с ревью-историей невозможен

**Priority:** P0
**Type:** Security (негативный)

### Objective
Убедиться, что пользователя с ревью-историей нельзя удалить физически: DELETE-эндпоинта нет (405), в UI нет кнопки удаления, единственный инструмент — soft-бан. Инвариант: ревью-таблицы ссылаются FK на `users.handle`.

### Preconditions
- Тест-стенд :3001, seed выполнен (`npm run test:reset`).
- Вход админом. reviewer имеет ревью-историю (primary у chp_published и chp_under_review, записи в `reviewer_history`/`chapter_reviewers`).

### Test Steps
1. Открыть карточку `/admin/users/reviewer` и осмотреть все действия.
   **Expected:** Кнопки «Удалить пользователя» НЕТ. Доступны только «Заблокировать (бан)», «Запретить комментарии», степпер «Ёмкость ревью».
2. Из консоли браузера (same-origin) выполнить:
   `fetch("/api/admin/users/reviewer", { method: "DELETE" }).then(r => r.status)`
   **Expected:** **405** (Method Not Allowed) — route-хендлер экспортирует только PATCH.
3. Открыть публичную страницу опубликованной главы `/blog/async-deep-dive/event-loop`.
   **Expected:** Кредит ревьюеров (reviewer и др.) отображается без изменений — ревью-история не тронута.
4. Убедиться, что soft-путь работает: на карточке reviewer нажать «Заблокировать (бан)», затем «Разблокировать».
   **Expected:** Оба действия успешны; запись пользователя и его ревью-история сохраняются.

### Test Data
- Handle: reviewer (пароль `password`, но вход не требуется).
- Глава с кредитом: chp_published (`event-loop`).

### Post-conditions
- Состояние seed восстановлено (reviewer разблокирован).

### Notes
- Инвариант CLAUDE.md: «пользователя с ревью-историей нельзя hard-delete (только soft/бан)» — `users.handle` UNIQUE и иммутабелен, на него ссылаются `chapter_reviewers`, `reviewer_history` и др.
- Любой появившийся в будущем DELETE-эндпоинт обязан быть закрыт этим кейсом (регресс-ловушка).
