# TC-AUTHOR — Тест-кейсы: Автор (handle `author`, блог `blog_async`)

**Кейсов: 25** (позитивных 17 · негативных 8)
**Легенда приоритетов:** P0 — security/data-loss/критический инвариант/основной путь публикации · P1 — основная фича роли сломана без workaround · P2 — частичная деградация с workaround · P3 — косметика.

Общие предусловия для всех кейсов: тест-стенд **:3001** (`npm run dev:test`), БД `blog.test.db`, seed выполнен (`npm run test:reset`). Если не сказано иное — выполнен вход как `author` (пароль `password`) через `/login`, кнопка «Войти». Никогда не выполнять кейсы на `:3000`.

Seed-контекст роли: блог `blog_async` (slug `async-deep-dive`, «Глубоко в асинхронность JavaScript», сложность medium, теги JavaScript/Async/Event Loop) с главами: `chp_published` (`event-loop`, «Цикл событий», published, v1+v2), `chp_under_review` (`promises`, «Промисы изнутри», under-review, primary `reviewer`), `chp_changes` (`async-await`, «Async/await на практике», changes-requested, primary `lena_review`), `chp_draft` (`generators`, «Генераторы и итераторы», draft, skills «Генераторы»/«Итераторы»). Чужой скрытый блог: `blog_ghost` (`hidden-blog`) с главой `chp_ghost` (`intro`).

## TC-AUTHOR-01: Кабинет /author — список блогов и агрегаты статусов глав

**Priority:** P1
**Type:** Functional

### Objective
Убедиться, что кабинет автора показывает его блоги с корректными счётчиками глав и индикацией статусов (published/under-review/changes-requested/draft).

### Preconditions
- Тест-стенд :3001, seed выполнен (`npm run test:reset`).
- Вход как `author`/`password`.

### Test Steps
1. Откройте `http://localhost:3001/author`.
   **Expected:** Eyebrow «Кабинет автора» + h1 «Мои блоги» (ui-feedback-4 П1, прототип author-portal.jsx). Сетка карточек (max 2 колонки): первой идёт плитка «Новый блог» (круг «＋»), далее карточка блога «Глубоко в асинхронность JavaScript». Справа aside 300px.
2. Осмотрите карточку блога.
   **Expected:** Сверху ряд из 4 точек-статусов (published — заполненная зелёная; on-review/changes/draft — контурные) + чип «ваш ход» (есть глава «нужны правки»); title + summary; строка «4 главы · 1 опубл.» с бейджами «1 на ревью» / «1 черн.»; футер: кнопка «＋ Глава» (outline-accent) + pin-тоггл 38px. Бейджа «Опубликован/Черновик» и кнопки «Открыть →» больше нет.
3. Кликните по телу карточки (title).
   **Expected:** Переход на `/author/blog/async-deep-dive` (деталь блога).
4. Вернитесь в кабинет и осмотрите aside: карточка «Об авторе» → «Изменить».
   **Expected:** Переход на `/author/portfolio`; глаз-тоггл переключает видимость портфолио без перехода. Ниже — «События» (последние уведомления с цветными маркерами).

### Test Data
- Блог: `blog_async` / `async-deep-dive`; главы `chp_published`, `chp_under_review`, `chp_changes`, `chp_draft`.

### Post-conditions
- Состояние БД не изменено.

### Notes
- Чужой блог `blog_ghost` в кабинете отсутствовать обязан (см. TC-AUTHOR-23).

## TC-AUTHOR-05: Редактор Variant B — слэш-меню «/» и сохранение

**Priority:** P1
**Type:** Functional

### Objective
Проверить добавление блоков через слэш-меню (параграф, заголовок, код) и явное сохранение черновика.

### Preconditions
- Тест-стенд :3001, seed выполнен; вход как `author`.
- Открыт редактор черновика (глава из TC-AUTHOR-04 или `/author/blog/async-deep-dive/generators/edit`).

### Test Steps
1. Введите заголовок главы «Черновик для теста» в поле «Заголовок главы».
   **Expected:** Индикатор в шапке меняется на «не сохранено» (жёлтая точка), «Сохранить» становится активной.
2. В пустом параграфе введите `/`.
   **Expected:** Появляется слэш-меню (listbox «Вставить блок») со списком типов: «Параграф», «Заголовок 2», «Заголовок 3», «Цитата», «Список», «Код», «Callout», «Схема Mermaid», «Изображение», «Таблица», «Embed».
3. Введите `/заг` и выберите «Заголовок 2»; введите текст «Раздел один».
   **Expected:** Меню фильтруется по подстроке; параграф заменён на блок «Заголовок 2» с текстом.
4. Добавьте блок «Параграф» с текстом «Первый абзац содержательного текста.» и через `/код` — блок «Код» с телом `console.log(1)`.
   **Expected:** Блоки добавляются по порядку; блок кода имеет моноширинное поле.
5. Нажмите «Сохранить».
   **Expected:** PATCH `/api/author/chapters/{id}` → 200; индикатор — «сохранено» (зелёная точка).
6. Обновите страницу (F5).
   **Expected:** Заголовок и все три блока на месте — контент персистентен.
7. Нажмите «Просмотр».
   **Expected:** Открывается `/author/blog/…/preview`; блоки отрендерены тем же рендерером, что и в ридере (заголовок, абзац, код с подсветкой).

### Test Data
- Блоки: h2 «Раздел один», p «Первый абзац содержательного текста.», code `console.log(1)`.

### Post-conditions
- Черновик изменён; сброс — `npm run test:reset`.

### Notes
- Ctrl/Cmd+S тоже сохраняет (проверить опционально). Rate-limit сохранений 1/сек — при слишком частых кликах «Сохранить» возможен 429 (см. TESTING.md §4).

## TC-AUTHOR-08: SubmitSheet — навыки (skills) обязательны, без них submit заблокирован

**Priority:** P0
**Type:** Functional / Negative
**Негативный:** да (блокировка гейтом + серверный отказ)

### Objective
Убедиться, что без ключевых навыков статьи отправка на ревью невозможна ни через UI (чек-лист), ни в обход UI (серверная перепроверка).

### Preconditions
- Тест-стенд :3001, seed выполнен; вход как `author`.
- Открыт редактор `/author/blog/async-deep-dive/generators/edit` («Генераторы и итераторы», draft; skills в seed заполнены — их нужно очистить в шаге 2).

### Test Steps
1. Нажмите «Отправить на ревью →».
   **Expected:** Открывается правая шторка «Отправить на ревью» с чек-листом «Готовность N/7»: «Заголовок главы (от 6 символов)», «Есть хотя бы один раздел (заголовок H2)», «Не меньше трёх содержательных блоков», «Хотя бы один тег блога», «Ключевые навыки статьи (1–6)», «Ревьюеров по сложности: 2–3», «Назначен ведущий из выбранных».
2. В секции «Ключевые навыки статьи (1–6)» удалите все чипы навыков («Генераторы», «Итераторы»).
   **Expected:** Пункт чек-листа «Ключевые навыки статьи (1–6)» становится незакрытым (○); внизу — подпись «Закройте все пункты»; кнопка «Отправить» неактивна (disabled), даже если выбрать ревьюеров и ведущего.
3. Во вкладке «По навыкам» осмотрите список ревьюеров.
   **Expected:** Пустое состояние «Добавьте навыки статьи — по ним подбираются ревьюеры.» — подбор без навыков не работает.
4. Обход UI: в DevTools Console выполните
   `fetch('/api/author/chapters/chp_draft/submit',{method:'POST',headers:{'Content-Type':'application/json'},body:JSON.stringify({skills:[],reviewers:['sergey_review','max_review'],primary:'sergey_review',note:'',complexity:'medium'})}).then(r=>r.status)`.
   **Expected:** 400 (гейт перепроверен на сервере; в теле — ошибка/`failedChecks` с пунктом про навыки). Статус главы в БД остаётся `draft`.
5. Обновите `/author/blog/async-deep-dive`.
   **Expected:** «Генераторы и итераторы» по-прежнему «Черновик» — никаких приглашений не создано.

### Test Data
- Глава `chp_draft` (slug `generators`); тело POST из шага 4.

### Post-conditions
- Если навыки были удалены и сохранены — восстановить через `npm run test:reset`.

### Notes
- Бизнес-правило (binding): submit требует заполненных skills; навыки статьи ≠ теги блога и ≠ компетенции ревьюера.

## TC-AUTHOR-09: Отправка на ревью создаёт pending-приглашения, а НЕ активных ревьюеров

**Priority:** P0
**Type:** Functional / Integration
**Акцент:** MATCH-CONSENT (согласие через приглашения)

### Objective
Проверить, что успешный submit переводит главу в «На ревью» и создаёт `review_invitations` (pending); ревью не стартует до accept (в `chapter_reviewers` никого нет).

### Preconditions
- Тест-стенд :3001, seed выполнен; вход как `author`.
- Открыт редактор `/author/blog/async-deep-dive/generators/edit` (черновик с валидным контентом: заголовок ≥6 символов, есть H2, ≥3 содержательных блока; при необходимости дополнить блоками как в TC-AUTHOR-05).

### Test Steps
1. Нажмите «Отправить на ревью →»; убедитесь, что навыки заполнены («Генераторы», «Итераторы»).
   **Expected:** Шторка открыта; вкладка «По навыкам» показывает «Ревьюеры под эти навыки не найдены.» (в seed нет ревьюеров с такими компетенциями).
2. Переключитесь на вкладку «Все», отметьте чекбоксами «Сергей Секьюрити» (`sergey_review`) и «Макс Девопс» (`max_review`); у Сергея нажмите «Сделать ведущим».
   **Expected:** Счётчик «Ревьюеры (2/3)»; у Сергея бейдж «ВЕДУЩИЙ»; чек-лист полностью закрыт — «Готово к отправке», кнопка «Отправить» активна.
3. Нажмите «Отправить».
   **Expected:** POST `/api/author/chapters/chp_draft/submit` → 200; редирект на `/author/blog/async-deep-dive`; глава «Генераторы и итераторы» — пилюля «На ревью».
4. Осмотрите строку главы в детали блога.
   **Expected:** Чипов «Команда:» НЕТ — активных ревьюеров ещё не существует (согласие не получено); есть только приглашения pending.
5. Выйдите (меню aria-label «Меню пользователя» → «Выйти»), войдите как `sergey_review`/`password`, откройте `/reviewer`.
   **Expected:** В инбоксе — новое приглашение на «Генераторы и итераторы» со статусом pending и действиями принять/отклонить; в очереди активного ревью главы НЕТ (вердикт/треды недоступны до accept).

### Test Data
- Глава `chp_draft`; ревьюеры `sergey_review` (ведущий), `max_review`; сложность блога medium → 2–3 ревьюера.

### Post-conditions
- Глава в `under-review`, созданы pending-приглашения. Сброс — `npm run test:reset`.

### Notes
- Downstream-гейты (verdict/threads/chat/publish) опираются на `chapter_reviewers` — до accept ревьюер не участник сессии.
- Ревьюеры со статусом «загружен» (full) не выбираются (чекбокс disabled) — проверить попутно.

## TC-AUTHOR-25: НЕГАТИВ — /admin и /reviewer недоступны автору (редирект)

**Priority:** P1
**Type:** Security / Negative
**Негативный:** да (307/redirect)
**Инвариант (TESTING.md §4):** редирект защищённых сегментов / гейтинг ролей

### Objective
Убедиться, что сегменты чужих ролей серверно редиректят автора (без клиентских заглушек).

### Preconditions
- Тест-стенд :3001, seed выполнен; вход как `author`.

### Test Steps
1. Откройте `http://localhost:3001/admin` (и `/admin/dashboard`).
   **Expected:** Серверный редирект (307) — автор попадает на `/` (для залогиненного не-админа admin-гейт ведёт на главную, не на `/admin/login`); admin-портал не рендерится даже частично.

   **Expected:** Редирект на `/` (не та роль); инбокс ревьюера не показан.

4. Контроль: откройте `/author`.

### Post-conditions
- Состояние БД не изменено.

### Notes
- Смежный инвариант «GET /admin,/author,/reviewer без cookie → 307 на логин» покрывается в TC-GUEST; здесь — вариант «есть сессия, но не та роль».
