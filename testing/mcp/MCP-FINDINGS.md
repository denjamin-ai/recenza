## 3. Ответы на 4 открытых вопроса плана

**(а) Статус PATCH чужой главы (403 vs 404).**
**404** — `PATCH /api/author/chapters/chp_ghost` → 404 `{"error":"Глава не найдена."}`. Ownership
маскируется под отсутствие ресурса, не 403. (Секция 03.)

**(б) UI publish-гейта (disabled vs отсутствие vs 403).**
Кнопка «Опубликовать» при неполных approve **отсутствует в DOM** (условный рендер
`allApproved && active` в `action-bar.tsx`) — вместо неё «Отправить v{N+1}» и пилюля «есть запрос
правок». `disabled` у неё бывает только на busy после клика. Серверный гейт: publish автором без всех
approve → **409** «Опубликовать можно только когда все ревьюеры одобрили.». (Секции 03, 04.)

**(в) Достижим ли match<50% на seed для позитивного flag-теста — у кого.**
Да, тривиально: для навыков «Генераторы, Итераторы» match = **0% у ВСЕХ четырёх** seed-ревьюеров
(Лена Базы 4.9 «загружен», Раиса Ревьюер 4.6 «занят», Макс Девопс 4.2 «свободен», Сергей Секьюрити 3.8
«свободен»); бейдж процента при 0 не рендерится, вкладка «По навыкам» пуста. Позитивный flag-тест
возможен с **любым** приглашённым (проще всего — сидовое `inv_pending`: sergey_review → «Промисы
изнутри», 0%). Негативный UI-тест «кнопка скрыта при match≥50%» на seed **невозможен** (нет ревьюера с
пересечением навыков) — серверный гейт `pct >= 50 → 409` покрыт только кодом роута. (Секции 03, 05.)

**(г) Целевой URL intent-replay.**
Значение `next` из URL логина — **возврат на главу** `/blog/async-deep-dive/event-loop`, не `/`.
Формат: `/login?next=<path>&intent=vote:<chapterId>:<1|-1>` либо `intent=bookmark:<blogId>`. Действие
реплеится автоматически (POST соответствующего API → 200) при переходе; тоста нет, состояние кнопок
обновлено. ⚠️ Реплей — toggle: может **снять** уже стоящий голос/закладку (см. баг №4). Ссылка
«Войдите» в комментариях несёт только `next=<путь>#comments` без intent. (Секция 01.)

## 5. Гочи для спеков (поведение, тексты, порядок действий)

### Auth и роли
- Логин-редиректы по ролям (`roleHome`): reader → `/`, author → `/author`, reviewer → `/reviewer`; `?next=` имеет приоритет. Гостевые редиректы `/author`, `/reviewer` → `/login` **без** `?next=`; `/admin/dashboard` → `/admin/login`. Reader на чужих protected — молча редирект на `/`.
- Сессии взаимоисключающие (isAdmin XOR userId): под админом `GET /api/auth/user` → `{"user":null}`; админ на author-API получает **401** («Требуется вход.»), не 403. Под admin-сессией `/author` → `/admin/dashboard`.
- `/login` при уже залогиненном НЕ редиректит (показывает форму). Выход из админки — «Выйти к блогу» (DELETE `/api/auth` + переход на `/`).
- Intent-replay = toggle (может снять seed-голос/закладку). Rate-limit логина считает только провалы — успешные логины в fixtures безопасны.

### Порядок действий review-flow
- Submit создаёт **приглашения** (pending), не назначение: команда пуста до accept; автору pending нигде не видно. После submit — редирект в кабинет блога (тост не успевает показаться).
- «Одобрить»/«Опубликовать» — без подтверждения; фидбек — тост `role=status` и смена пилюль. Force-approve — инлайн-подтверждение «Опубликовать в обход гейта?» (не `confirm()`).
- «Отправить v{N+1}» доступна и на «На ревью». После v2: вердикты обнулены, треды пер-ревизионные, carry-forward без re-consent, «все одобрили» = approve каждого перенесённого.
- «Сменить ведущего» disabled при <2 принявших; смена только через админа. Кандидаты — только принявшие ревьюеры.
- Editor lock: under-review/published — read-only баннер, API PATCH → 409 «Главу нельзя редактировать в текущем статусе.»; changes-requested редактируется полноценно.
- «загружен» (full) ⇒ чекбокс ревьюера disabled; accept при full → 409. При понижении сложности лишние выбранные ревьюеры срезаются.
- Flag «навыки не совпадают» — без подтверждения; сервер перепроверяет match<50% (иначе 409); остальные pending-приглашения ревизии гасятся в declined.

### Комментарии
- Глубина от 0, максимум 2; ответ на узел глубины 2 → 409. Anchor только у root (у ответов null), quote ≤600 символов. Плавающая кнопка якоря гаснет на scroll/resize/Escape — не скроллить между выделением и кликом.
- Удаление — native `confirm("Удалить комментарий?")`; tombstone только при живых потомках (seed `cmt_deleted` невидим — для теста tombstone создавать ветку с ответом).
- Пустой POST проверяется только fetch'ем (кнопка disabled) → 400 «Пустой комментарий.». Голос за своё → 403. Двойной клик гасится busy-guard (один POST); 429 — только параллельными fetch.
- Ревизия комментария штампуется сервером; относительное время («2 дн назад») — не ассертить точные строки.

### Тексты тостов и ошибок (точные строки)
- Тосты: «✓ Вы одобрили главу.» · «✓ Запрос на смену ведущего отправлен админу.» · «Запрос отправлен админу. Статус — в кабинете автора.» · «Заявка отправлена! Администратор её рассмотрит.»
- Ошибки API: «Требуется вход.» (401) · «Недостаточно прав.» (403) · «Неверный никнейм или пароль.» (401) · «Слишком глубокая вложенность.» (409) · «Окно редактирования истекло.» (403) · «Нельзя редактировать чужой комментарий.» (403) · «Нельзя голосовать за свой комментарий.» (403) · «Комментирование ограничено.» (403) · «Ревьюеры не участвуют в публичных обсуждениях.» (403) · «Слишком часто. Подождите секунду.» (429, `Retry-After: 1`) · «Главу нельзя редактировать в текущем статусе.» (409) · «Глава не найдена.» (404) · «Опубликовать можно только когда все ревьюеры одобрили.» (409) · «На приглашение уже дан ответ.» (409) · «Укажите причину отклонения.» (400) · «Внешний баннер: укажите http(s)-ссылку.» (400) · «Пустой комментарий.» (400)

### Прочее
- Публичный профиль — по **slug** (`/u/sergey-review`); `/u/{handle}` с подчёркиванием → 404.
- Два `nav[aria-label="Навигация по блогу"]` — strict mode violation, скоупить.
- Console-мониторинг: allowlist на 404 `/uploads/*` (`articles/event-loop.png`, `covers/async.png`, `donations/sbp-qr.png` — файлов нет до Фазы 12) и dev-шум turbopack (preload CSS).
- Оценка ревьюера: агрегат пересчитывается по реальным строкам `reviewer_ratings` — seed-значение «3.8 (3)» (синтетическое) перезаписывается («★ 4.0 · 1 оценок»); не ждать усреднения с seed-числом. Секция «Оцените ревьюеров» на seed скрыта (все оценены) — для теста нужна свежая публикация.
- Recruit: дедуп pending на (главу, автора) → повторный POST 200 `{alreadyPending:true}` (не ошибка). Приём заявки с доски меняет роль → после accept обязателен `test:reset`.
- Админ: «удалить» у баннеров и board-calls — **без подтверждения**; поиск юзеров только по submit; donate-эффекты гость видит после перезагрузки (RSC-пропсы).
- Title как маркер работает только у `/author` («Кабинет автора | Recenza») и логин-страниц; `/reviewer` и админка держат общий title.
- Слэш-меню и «+ Блок» — 11 пунктов (вариации list/callout не развёрнуты; расхождение с «14 типов» спеки — учитывать в asserts).
- «Мёртвые» trusted-клики MCP: в спеках после клика ждать сетевой ответ/aria-pressed/URL и ретраить; первым делом в 11.3 проверить, что в чистом `@playwright/test` клики доставляются (иначе — блокер, копать окружение).

## 6. Баги и аномалии (сводно, по severity)

### P1 — сценарные пробелы (ломают ожидаемый сценарий)
1. **Publish не уведомляет подписчиков автора** — `createNotifications` в `src/app/api/review/[chapterId]/publish/route.ts` шлёт только команде ревью; подписанный reader не получил «Новая глава» ни за одну публикацию. Из-за этого PUB-ARTICLE = ⚠️. (С4)
2. **Force-approve не закрывает pending `primary_change_request`** — `pcr_1` остался pending, счётчик «Смена ведущего: 1» на дашборде ведёт в пустую очередь; разобрать запрос через UI негде. (С6)
3. **Снятие ведущего ревьюера не переназначает primary** — `chapters.primary_handle` остался на снятой lena_review, у оставшегося `is_primary=0`: dangling primary. (С6)

### P2 — заметные дефекты поведения/UX
4. **Intent-replay = toggle**: реплей `vote:chp_published:1` под reader со стоящим seed-голосом **снял** голос (2→1) вместо «поставить»; то же грозит `bookmark:blog_async`. (С1)
5. **Pending-приглашения невидимы автору**: после submit команда пуста, отменить/повторить приглашение из UI нельзя — только уведомления accept/decline. (С4)
6. **Markdown-шорткат «## » теряет фокус и текст**: блок конвертируется в «Заголовок 2», но фокус не переносится в новый INPUT — набранное после «## » уходит в никуда (`applyShortcut`, `markdown.ts` / `block-list-editor.tsx:82`). (С3)
7. **Уведомление comment_reply некликабельно**: seed-payload содержит только `commentId`, а `NotificationBell.hrefOf()` требует `payload.href` — клик лишь помечает прочитанным. (С2)
8. **Дубли pending-запросов смены ведущего**: seed `pcr_1` + новый видны админу одновременно; утверждение устаревшего (from уже не ведущий) проходит 200 без ошибки. (С4, С6)

### P3 — минорные / косметика
9. Гостевые редиректы `/author`, `/reviewer` → `/login` **без** `?next=` — диплинк теряется. (С1)
10. Title не обновляется на `/reviewer` и всех admin-страницах (общий дефолт). (С1, С6)
11. Два `nav[aria-label="Навигация по блогу"]` (mobile+desktop) — strict mode violation. (С2)
12. Textarea правки комментария без accessible name. (С2)
13. Dirty-индикатор «не сохранено» не сбрасывается при возврате значения к исходному. (С3)
14. Нет per-user дедупа заявок с доски — у reader две pending-заявки одновременно (кандидат в backlog; админка ведёт себя корректно). (С5)
15. Косметика сидового flag-алерта: двойная точка/регистр «…Навыки не совпадают (match < 50%).. Исправьте…». (С5)
16. Потеря кликов до гидрации (`/login` и «Отправить на ревью →» в редакторе) — клики молча съедаются; тест-риск флаки. (С3, С4)

### Seed / данные (не код рантайма)
17. 404 на seed-ассеты: `/uploads/articles/event-loop.png`, `/uploads/covers/async.png`, `/uploads/donations/sbp-qr.png` (битая картинка QR в DonateModal) — console error на многих страницах; файлов нет до Фазы 12 → allowlist. (С1–С6)
18. Seed-summary опубликованных глав выглядит странно в ленте: «Промисы изнутри» — «Отправлено на ревью, ожидает вердиктов.», «Генераторы» — «Черновик, ещё не отправлен.» (данные, не рендер). (С4, С6)

### Среда / инфраструктура MCP (не приложение)
19. **Интермиттентная потеря trusted-кликов/клавиш Playwright MCP** (клик «выполняется», но событие не доходит до React; `el.click()` через evaluate работает всегда) — воспроизводилось в С2, С5, С6. **Перепроверить первым делом в чистом `@playwright/test` Фазы 11.3** — если воспроизведётся, это блокер автотестов. (С2, С5, С6)
20. MCP-профиль браузера хранит cookie между сессиями (стейл-admin-сессия в начале С1). В автотестах контексты чистые; при MCP-исследовании сначала гасить сессии (`DELETE /api/auth`, `DELETE /api/auth/user`). (С1)
