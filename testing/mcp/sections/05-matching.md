## Ответы на вопросы

1. **У кого match < 50% для навыков «Генераторы, Итераторы»?** — У ВСЕХ четырёх seed-ревьюеров match = 0%
   (бейдж процента вообще не рендерится при 0): Лена Базы (Базы данных/SQL/Производительность),
   Раиса Ревьюер (TypeScript/React/Архитектура/Event Loop), Макс Девопс (DevOps/Docker/CI/CD),
   Сергей Секьюрити (Безопасность/Криптография). Вкладка «По навыкам» пуста → «Ревьюеры под эти навыки
   не найдены». Позитивный flag-тест возможен с ЛЮБЫМ приглашённым; негативный UI-тест «кнопка скрыта при
   match ≥ 50%» на seed-данных НЕВОЗМОЖЕН (нет ревьюера с пересечением навыков) — серверный гейт
   покрывается только кодом (`pct >= 50` → 409 в `api/reviewer/invitations/[invitationId]`).
2. **Сортировка «Топ» (вкладка «Все»)** при 0% у всех определяется рейтингом/объёмом: Лена 4.9 (загружен,
   чекбокс disabled) → Раиса 4.6 (занят) → Макс 4.2 (свободен) → Сергей 3.8 (свободен).
3. **Кнопка запроса админу** — не отдельная страница, а кнопка «Запросить ревьюеров у админа» в пустом
   состоянии вкладки «По навыкам» SubmitSheet (появляется при навыках ≥1 и нуле совпадений; после отправки
   заменяется текстом успеха). Дедуп: повторный POST при pending-запросе на ту же главу → `{ok, alreadyPending:true}`.
4. **Заявки с доски разбираются там же**: /admin/recruit, секция «Отклики с доски · на рассмотрении N»
   (отдельного раздела нет).

## Тайминги

- `npm run test:reset` ≈ 3–5 с; безопасен при живом dev-сервере.
- Все ключевые роуты были уже скомпилированы (долгоживущий dev-сервер) — навигации ≤1–2 с; на свежем
  сервере первая компиляция роута занимает секунды (в спеках держать увеличенный timeout первой загрузки).
- Мутации (submit/accept/verdict/publish/ratings/recruit/applications) — API < 500 мс, но обновление UI —
  через `router.refresh()` (RSC-перезагрузка): состояние на экране меняется через ~1–2 с. В спеках ждать
  ТЕКСТ результата (`все одобрили`, «Новых приглашений нет.»), не response.
- Подбор в SubmitSheet пересчитывается на клиенте мгновенно при правке навыков (чистый `rankReviewers`).
- Кросс-экранный sync ревью — поллинг 30 с; для мгновенной проверки использовать перезагрузку страницы.

## Поведение и гочи

- **Consent-модель работает как заявлено**: submit создаёт только `review_invitations` (pending);
  на карточке главы автора команда ПУСТАЯ, на ревью-странице «Команда ревью» пустая и «0 реценз.»,
  pending-приглашение автору нигде явно не показывается (нет UI «ожидает согласия») — только после
  accept появляются аватар/вердикты.
- SubmitSheet: гейт чек-листа = 7 пунктов; «Не меньше трёх содержательных блоков» — сидовый черновик
  `generators` (H2+абзац) НЕ проходит, нужно добавить блок. Максимум ревьюеров зависит от сложности
  (Простая 1–2 / Средняя 2–3 / Сложная 3–5); при понижении сложности лишние выбранные срезаются.
- Ревьюер с `full` (Лена, load≥capacity) — чекбокс disabled (не выбрать); accept при full → 409 (защита от гонок).
- Flag: подтверждающего диалога НЕТ — клик сразу шлёт `{action:'flag'}`; сервер перепроверяет match<50%
  (иначе 409). Побочные эффекты: ревизия → `changes-requested`, остальные pending-приглашения ревизии
  гасятся в declined, уже принявшие остаются (код роута).
- Decline: `chapter_reviewers`/`reviewLoad` не трогаются; автору уведомление «Ревьюер отклонил приглашение: {глава}».
- Уведомления автора: типы `review_invite_declined`, `review_skills_mismatch` (payload.href →
  `/author/blog/{slug}/{chapter}/review`), `review_invite_accepted` при accept.
- Оценка: POST `/api/author/ratings` `{chapterId, reviewerHandle, stars}`; после оценки агрегат
  пересчитывается по РЕАЛЬНЫМ строкам `reviewer_ratings`: у Сергея seed-значение «3.8 (3)» (синтетическое,
  без строк) сменилось на «★ 4.0 · 1 оценок» на публичном профиле. В спеках не ждать усреднения с seed-числом.
- Публичный профиль — `/u/{slug}` (slug с дефисом: `sergey-review`); `/u/{handle}` с подчёркиванием — 404.
- Recruit: дедуп «один pending на (главу, автора)» → повторный POST отвечает 200 `{alreadyPending:true}`
  (не ошибка). Одобрение админом — двухшаговое (Одобрить → форма → «Опубликовать на доске»), направление
  сразу видно гостю на /board (фильтр-чип навыка тоже появляется).
- Заявка с доски: у залогиненного юзера поле «Имя» скрыто; заявка от reader с уже существующей pending-заявкой
  (seed Frontend) создаётся ВТОРАЯ (дедупа per-user нет — дедуп, видимо, per-направление). После выдачи роли
  оставшаяся pending-заявка того же юзера показывает кнопку «Принять» вместо «Принять (выдать роль)».
- Принятие заявки пишет навыки заявки в `users.competencies` нового ревьюера.
- Rate-limit на ответ по приглашению (`invite-respond`) и recruit-request — 1/сек → 429 c `Retry-After`;
  в спеках между мутациями одной роли вставлять ≥1 с или ждать UI.
- Логин-редирект по роли (`roleHome` в `login-form.tsx`): author → `/author`, reviewer → `/reviewer`, читатель → `/`.
- Кнопки админа: `POST /api/admin/recruit-requests/{id}`, `POST /api/admin/applications/{id}` — оба 200 + RSC refresh.

## Баги/аномалии

1. **[Среда/MCP, не приложение] Потеря trusted-кликов в длинной MCP-сессии.** Начиная с экрана редактора,
   значительная часть `browser_click` (и Enter в форме логина) выполнялась Playwright'ом без ошибки, но
   события не доходили до React-обработчиков (network-запросов нет, состояние не меняется); `elementFromPoint`
   подтверждал кликабельность, повторный клик иногда срабатывал. Programmatic `el.click()` через
   `browser_evaluate` срабатывал всегда (тот же обработчик/API). Похоже на деградацию доставки input-событий
   долгоживущей MCP-сессии, НЕ на баг приложения (логин-клик в начале сессии работал). Для Фазы 11:
   в TS-автотестах с чистым `@playwright/test` перепроверить — если воспроизведётся, это станет блокером.
2. **/u/{handle} → 404** (`/u/sergey_review`), работает только slug `/u/sergey-review`. Не баг (роут по slug),
   но ловушка для спек: в UI handle показывается с `@`, а ссылки строятся по slug.
3. **Косметика текста flag-алерта**: при сидовом flagged (`async-await`) в кабинете автора двойная точка и
   смешение регистров — «Глава снята с ревью: Навыки не совпадают (match < 50%).. Исправьте…» (reason
   вставляется с собственной точкой). У живого flag reason по умолчанию «навыки не совпадают» — строка чистая.
4. **Консоль**: повторяющийся 404 `/uploads/articles/event-loop.png` (сид-обложка без файла — известно) —
   попадает в console.error на многих страницах; учесть в мониторинге консоли спеками (allowlist).
5. Дедуп заявок с доски отсутствует per-user: reader смог иметь две pending-заявки одновременно (seed Frontend
   + новая Backend). Если это не задумано — кандидат в backlog; поведение админки корректное.
