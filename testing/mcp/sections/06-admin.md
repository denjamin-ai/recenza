## Тайминги

- Dev-сервер :3001 был прогрет предыдущими кластерами — рекомпиляций почти не было: admin-роуты открывались за ~1–2 с первый раз, <1 с повторно. `npm run test:reset` (~5–8 с) не инвалидирует компиляцию.
- Мутации (PATCH/POST + `router.refresh()`) отражаются в DOM за ~0.5–2 с — в спеках ждать `expect(...).toBeVisible()`/`toHaveText()`, не фиксированные таймауты.
- Карусель промо авто-ротируется (слайд «Поддержать» доступен не всегда) — перед кликом обязательно выбрать слайд точкой «Баннер N» и кликать сразу (иначе flaky).

## Поведение и гочи

- `/admin` → 307 на `/admin/dashboard`. Гейтинг страниц: залогиненный НЕ-админ на `/admin/**` → redirect `/`; гость → redirect `/admin/login` (см. `requireAdminPage`).
- Логин админа — POST `/api/auth` `{password}` → `{isAdmin:true}`; logout — DELETE `/api/auth`. Кнопка «Выйти к блогу» делает DELETE + `location.assign('/')`.
- Глобальный поиск в шапке админки работает **только по submit** (Enter) и всегда ведёт на `/admin/users?q=…` — live-фильтрации нет.
- Force-approve: подтверждение инлайновое (не `confirm()`), текст «Опубликовать в обход гейта?». После публикации карточка исчезает из очереди; reviewLoad −1, `reviewer_history` кредитуется, автору уходит notification `force_approved`.
- Снятие ревьюера: причина необязательна; приглашение гасится переводом в `declined`; `removed_reviewers` — лог для админа; повторно ревьюер в списке не отображается.
- Recruit reject: двойная защита причины — disabled-кнопка на клиенте + 400 на сервере.
- Банер-«удалить» и board-call-«удалить» — **без подтверждения** (мгновенный DELETE) — учесть в тестах (не кликать лишний раз).
- Донат-флаг живёт в `app_settings` (`donations_enabled`), тумблер шлёт `{donationsEnabled:boolean}`; способы передаются в модалку через RSC-пропсы (перезагрузка страницы нужна, чтобы гость увидел изменение).
- Статусы API (сводка): PATCH users 200 · PATCH reports 200 · force-approve 200 · remove-reviewer 200 · recruit reject-no-reason **400** · banners XSS-target **400** · DELETE users **405** · author-API под админом **401** · admin-API под reader **403**.
- Заявка с доски в seed — от `@reader` («Рина Читатель»), не `app_user`. Приём заявки меняет роль → в тестах после accept обязателен reset.
- У карточки troll после включения комментариев кнопка меняется на «Запретить комментарии» — тумблер вернул в исходное состояние (эффект «troll может комментировать» до конца не доводился).

## Ответы на вопросы

- **Текст ошибки валидации target (external + `javascript:alert(1)`)**: HTTP **400**, в форме появляется `role=alert` с текстом **«Внешний баннер: укажите http(s)-ссылку.»** Сохранения не происходит.
- **Причина при reject recruit-запроса обязательна?** Да: кнопка «Отклонить с причиной» disabled при пустом поле; сервер отвечает 400 «Укажите причину отклонения.»
- **Где ведутся board-calls**: на `/admin/recruit`, секция «Доска "Ищем ревьюеров"» (создание/«срочно»/удаление там же).
- **Donate-баннер при выключенных пожертвованиях**: баннер остаётся в карусели, модалка открывается, но показывает заглушку «Способы поддержки пока не настроены.»
- **Негативы**: админ → `POST /api/author/blogs` = **401** («Требуется вход.», а не 403 — у админ-сессии нет userId); reader → `GET /admin/users` = redirect на `/`; reader → `POST /api/admin/banners` = **403** («Недостаточно прав.»).

## Баги/аномалии

1. **[БАГ] Force-approve оставляет «висячий» запрос смены ведущего.** `pcr_1` (chp_under_review) после force-approve остался `pending`; на дашборде счётчик «Смена ведущего: 1» ведёт в `/admin/review`, но карточка главы из очереди исчезла — разобрать запрос через UI больше негде. Ожидание: публикация должна закрывать/отклонять pending primary-change-запросы.
2. **[БАГ/аномалия] Снятие ведущего ревьюера не переназначает primary.** После снятия lena_review с `chp_changes` поле `chapters.primary_handle` осталось `lena_review` (снятой), у оставшегося ревьюера `is_primary=0` — глава остаётся без действующего ведущего, dangling-ссылка на снятого.
3. **[Данные/ассет] QR метода «СБП» отдаёт 404** — `/uploads/donations/sbp-qr.png` не существует (console error на модалке «Поддержать»); битая картинка у гостя. (Загрузка файлов — Фаза 12, но seed ссылается на несуществующий файл.)
4. **[Minor/SEO] У всех admin-страниц дефолтный `<title>`** («Recenza — девблог с редакционным ревью») — нет «Пользователи | Recenza» и т.п.; у `/admin/login` title есть.
5. **[Среда, не приложение] Физические клики/клавиши Playwright MCP не доставлялись странице** (browser_click/press_key «выполнялись», но события не доходили — ни network-запросов, ни навигации; `element.click()` через evaluate работает, форма логина через JS-click работает). Похоже на квирк CDP-сессии на этой машине. Для Фазы 11 перепроверить обычным `@playwright/test` — если воспроизведётся, копать окружение (DPI/оффскрин-окно), это заблокирует автотесты.
6. **[Данные] Карточка «Промисы изнутри» в ленте после публикации показывает summary «Отправлено на ревью, ожидает вердиктов.»** — текст ревизии из seed, для читателя выглядит странно (не баг кода, но повод поправить seed).
