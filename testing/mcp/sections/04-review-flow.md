## Пройденные сценарии

- ✅ **REV-CHAPTER (полный цикл draft → published, глава generators/chp_draft)**
  1. author: редактор → «Отправить на ревью →» → SubmitSheet (dialog «Отправка на ревью»). Чек-лист «Готовность N/7»; для отправки добавил 3-й блок (было 2 → пункт «Не меньше трёх содержательных блоков»), сложность «Простая» (1–2 ревьюера), выбрал Раису («Все» таб), «Сделать ведущим» → «ВЕДУЩИЙ» [pressed] → «Готово к отправке» → «Отправить».
  2. После отправки: **редирект в кабинет блога** `/author/blog/async-deep-dive`, глава «На ревью», тост не зафиксирован. Строки «Команда:» у главы нет (никто не принял).
  3. reviewer: инбокс `/reviewer` — карточка «Входящие приглашения»: «Генераторы и итераторы», «…· v1 · ведущий», «0% совпадение», чипы навыков; кнопки «Принять»/«Отклонить»/«Навыки не совпадают» (последняя видна при match<50%). «Принять» → приглашение исчезло, глава в «Активных ревью» («вы ведущий · ваш ход · На ревью»), плитка «Активные ревью» 2→3.
  4. `/reviewer/review/chp_draft`: JS-выделение текста в article → плавающая кнопка «Прокомментировать» → композер панели переключился в якорный режим (тумблер «Комментарий» [pressed] / «Правка»). «Правка» → блок «Было» с фрагментом + поле «Как должно стать — замена выделенного…» + кнопка «Предложить». Тред создан: bauble «правка предложена: 1 тред(ов)…» на блоке, карточка треда справа (badge «правка», diff from/to, «Отметить решённым»).
  5. Клик по bauble → тред активируется в панели (композер: «↳ ответ @reviewer», placeholder «Ответить или дополнить…»). Кнопка «→ блок» в карточке — обратный переход к блоку.
  6. author (review-страница): тред виден, вместо «Отметить решённым» — **«Применить и закрыть»** → текст блока изменился in-place («…ставить выполнение функции на паузу…»), тред resolved (bauble «решено: 1 тред(ов)», счётчик «1 решено», «показать решённые» → карточка с badge «решено», кнопка становится «скрыть решённые» [pressed]).
  7. reviewer: «Одобрить» — **без подтверждения**, появилась пилюля «все одобрили».
  8. author: появилась кнопка «Опубликовать» (её нет до полного approve) → клик без подтверждения → пилюля статуса «Опубликовано», action-bar «Глава опубликована.», кнопки Опубликовать/Отправить v2 исчезли.
  9. гость `/blog/async-deep-dive/generators`: глава published, region «Ревьюеры главы» → «Эту версию проверяли»: «Раиса Ревьюер» + бейдж «ведущий»; «Ключевые навыки главы» видны.
- ✅ **REV-SESSION-CHAT (promises/chp_under_review)**: reviewer — «Развернуть чат сессии» (кнопка-аккордеон «Чат сессии N» ▸/▾), 3 seed-сообщения; textbox «Сообщение в чат сессии» (placeholder «Сообщение команде…»), кнопка «Отправить сообщение» (→, disabled пока пусто). Отправил → «Чат сессии 4». author (свежая загрузка страницы) — сообщение видно, ответил → «Чат сессии 5». Треды не изменились («Обсуждения 2») — **чат и треды не смешиваются** (review_chat ≠ threads).
- ✅ **REV-PRIMARY (promises)**: «Сменить ведущего» — у автора; **disabled при <2 принявших ревьюеров** (на generators c 1 ревьюером — disabled). Клик → модалка dialog «Сменить ведущего ревьюера»: текст «Запрос уйдёт админу. До его решения ведущим остаётся @reviewer.», radio-список кандидатов (Лена Базы @lena_review), textarea «Причина» (placeholder «Например: текущий ведущий недоступен.», обязательна — «Отправить запрос» disabled без неё), «Отмена»/«Отправить запрос». Тост (role=status): «✓ Запрос на смену ведущего отправлен админу.» Смена НЕ мгновенная — через `primary_change_requests`. Админ `/admin/review` («Ревью глав»): блок «Запрос смены ведущего: @reviewer → @lena_review», кнопки «Утвердить смену»/«Отклонить» (+ «Force-approve (опубликовать)», список ревьюеров с «снять»). «Утвердить смену» → POST `/api/admin/review/chp_under_review/primary` 200, Лена стала «ведущий». Инбокс lena_review: «Промисы изнутри … **вы ведущий**». ⚠️ Был и seed-запрос pcr_1 с тем же содержимым — второй (устаревший) утвердился тоже с 200 без ошибки.
- ✅ **PUB-CHAPTER-V2 + REV-VERSIONS (async-await/chp_changes)**: changes-requested глава **редактируется** обычным редактором (шапка «…· нужны правки», Сохранить/Отправить доступны). Правка сохранена → review-страница автора: action-bar «0 открытых · 2 реценз.» + пилюля «есть запрос правок», кнопка **«Отправить v2»** (в action-bar review-страницы; присутствует и на «На ревью»-главах, не только changes-requested). Клик → rev 2, статус «На ревью», **вердикты обнулены** (пилюля «есть запрос правок» исчезла), треды/решено v1 в панели пропали (пер-ревизионные), кнопка стала «Отправить v3». Carry-forward: команда (lena ведущий + reviewer) перенесена без повторного приглашения. lena «Одобрить» (тост «✓ Вы одобрили главу.») — «все одобрили» НЕ появилось: **нужен approve всех перенесённых** → reviewer тоже «Одобрить» → «все одобрили» → author «Опубликовать» → «Глава опубликована.». Ридер: «Эту версию проверяли»: Лена Базы (ведущий) + Раиса Ревьюер. Прошлых версий у async-await нет (v1 не публиковалась). Раскрытие «Прошлые версии» проверено на event-loop: group → «к версии v1» + список (Лена Базы…).
- ✅ **REV-WHOLE-BLOG**: strip глав на ReviewPage = `tablist "Главы блога"` с табами «01 Цикл событий», «02 Промисы изнутри», «03 Async/await на практике», «04 Генераторы и итераторы» ([selected] у текущей). Клик по табу = **навигация на review-URL той главы** (`/author/blog/…/generators/review`), контекст ревью сохраняется (тот же макет, своя команда/треды/чат). Тумблера «Весь блог» на ReviewPage **нет** — режим только в ридере: group «Режим чтения» → ссылки «Глава» / «Весь блог» (`?mode=whole`); в whole-режиме все published-главы рендерятся последовательно.
- ✅ **PUB-ARTICLE**: обе опубликованные главы видны гостю в ленте `/` (карточки article с link «Блог + Глава»). ⚠️ Уведомление подписчику — НЕ пришло (см. Баги).

## Тайминги

- `npm run test:reset` ≈ 5–10 с (безопасно при живом сервере).
- Все роуты были уже скомпилированы прошлыми сессиями dev-сервера — навигация <1 с; на холодном сервере первая компиляция роута занимает секунды (учитывать в first-run E2E).
- ⚠️ **Гидрация редактора**: клик по «Отправить на ревью →» сразу после загрузки страницы ТЕРЯЕТСЯ (первый клик не открыл SubmitSheet, второй открыл). В спеках — ждать гидрации (например, `expect(кнопка).toBeEnabled()` недостаточно; надёжнее дождаться интерактивности, повторить клик или подождать сетевой idle).
- Бейдж уведомлений появляется с задержкой (клиентский fetch `/api/notifications` после загрузки): сразу после load — «Нет новых уведомлений», через мгновение — счётчик.
- Кросс-экранный sync ревью — поллинг 30 с + router.refresh; в тестах надёжнее reload.

## Поведение и гочи

- **Submit создаёт приглашения, не назначение**: после «Отправить» команда ревью пуста; ревью стартует только после accept ревьюером. Автору pending-приглашение НИГДЕ не показывается (см. Баги). Автор узнаёт из уведомлений «Ревьюер принял/отклонил приглашение».
- После submit — **редирект в кабинет блога** (`/author/blog/{slug}`), не остаёмся в редакторе.
- «Лена Базы» в пикере disabled (checkbox) при статусе «загружен» (reviewLoad=capacity).
- Вердикты и publish — **без диалогов подтверждения**; фидбек — тост (role=status) + смена пилюль.
- «Отправить v{N+1}» доступна автору и в статусе «На ревью» (не только «Нужны правки»).
- После submit-revision: вердикты обнулены, треды пер-ревизионные (панель v2 пустая, «решено» v1 не переносится), команда carry-forward (re-consent не нужен) — «все одобрили» требует approve **каждого** перенесённого ревьюера.
- «Сменить ведущего» = запрос админу (`primary_change_requests`), не мгновенная смена; до решения админа ведущий прежний. Кандидаты в модалке — только принявшие ревьюеры главы.
- Публичный профиль ревьюера — по slug портфолио (`/u/lena-review` → 200; `/u/lena_review` → 404).
- Editor lock: PATCH `/api/author/chapters/chp_under_review` → **409** `{"error":"Главу нельзя редактировать в текущем статусе."}`; PUT → 405 (только PATCH). UI: баннер + скрытые кнопки.
- Негативные статусы (same-origin fetch): вердикт гостем → **403**; вердикт неназначенным (max_review) → **403**; вердикт автором → **403**; publish ревьюером → **403**; publish автором без всех approve → **409** `{"error":"Опубликовать можно только когда все ревьюеры одобрили."}`.
- API-карта flow: submit `POST /api/author/chapters/{id}/submit`; accept/decline/flag `POST /api/reviewer/invitations/{id}`; треды `POST /api/review/{chapterId}/threads`; apply `POST /api/review/threads/{threadId}/apply`; resolve `…/resolve`; чат `POST /api/review/{chapterId}/chat`; вердикт `…/verdict`; publish `…/publish`; v2 `…/submit-revision`; смена ведущего `…/primary-change`; админ-утверждение `POST /api/admin/review/{chapterId}/primary`.
- Review-страница автора и ревьюера — один макет, разные действия (author: Применить и закрыть/Опубликовать/Отправить vN/Сменить ведущего; reviewer: Отметить решённым/Нужны правки/Одобрить).
- В ленте карточка главы показывает `summary` главы как есть — у только что опубликованных «Генераторов» остался сид-текст «Черновик, ещё не отправлен.» (авторские данные, не рендер-баг).

## Ответы на вопросы

- **Где автору видно pending-приглашение?** Нигде явно: «Команда ревью» пуста, на карточке главы нет «Команда:». Только уведомления при accept/decline (типы review_invite_accepted/declined).
- **Как создаётся тред?** Выделение текста в article → плавающая кнопка «Прокомментировать» → композер панели в якорном режиме; тумблер «Комментарий»/«Правка»; «Правка» = suggestion с «Было» (фрагмент) и полем «Как должно стать…», кнопка «Предложить».
- **Клик по bauble открывает тред?** Да — активирует его в правой панели (композер переходит в режим ответа «↳ ответ @handle»). Обратно: кнопка «→ блок» в карточке треда.
- **«Одобрить» — есть подтверждение?** Нет, сразу; тост «✓ Вы одобрили главу.».
- **changes-requested редактируется?** Да, полноценный редактор (в отличие от under-review/published — там 409 и баннер).
- **Где «Отправить v2»?** В action-bar review-страницы автора (не в редакторе).
- **Вердикты обнулились после v2?** Да (пилюля «есть запрос правок» исчезла, «все одобрили» требует новых approve обоих).
- **Тумблер «Весь блог» на review?** Нет; только в ридере (group «Режим чтения»: «Глава»/«Весь блог» `?mode=whole`).
- **Чат не смешивается с тредами?** Не смешивается: отдельный аккордеон и счётчик «Чат сессии N», треды («Обсуждения N») не меняются.

## Баги/аномалии

1. **[БАГ/пробел] Publish не уведомляет подписчиков.** `POST /api/review/{id}/publish` создаёт уведомления только команде ревью (`createNotifications` по handles ревьюеров — `src/app/api/review/[chapterId]/publish/route.ts:114-130`). reader (follows: reader→author в сиде) не получил «Новая глава» ни за одну из двух публикаций, хотя тип уведомления существует (сид «Новая глава: Цикл событий»). Сценарий PUB-ARTICLE «подписчик узнаёт о новой главе» не работает.
2. **[UX/тест-риск] Потеря клика до гидрации.** Первый клик по «Отправить на ревью →» сразу после загрузки редактора не открыл SubmitSheet (обработчик ещё не навешан); второй клик открыл. Для e2e — обязательное ожидание интерактивности.
3. **[UX-пробел] Pending-приглашения невидимы автору.** После submit автор не видит, кого пригласил и ждёт ли ответа (команда пуста, карточка главы без команды); отменить/повторить приглашение из UI нельзя.
4. **[Edge] Дубли запросов на смену ведущего.** Возможны два одинаковых pending-запроса (`pcr_1` из сида + новый) — оба видны админу одновременно; утверждение второго (устаревшего: from уже не ведущий) проходит 200 без ошибки/предупреждения. Идемпотентно по результату, но список не схлопывается.
5. **[Извеcтный пробел сида] 404 на статику** `/uploads/articles/event-loop.png`, `/uploads/covers/async.png`, `/uploads/donations/sbp-qr.png` — console-ошибки на многих страницах (загрузки нет до Фазы 12). Учитывать при мониторинге console в тестах (фильтровать 404 uploads).
6. **[Замечание] Тост после submit не зафиксирован** — мгновенный редирект в кабинет блога; в alert-регионе только route-announcer. Если тост задуман — он не успевает показаться.
